## Архитектурные правила (НЕ нарушать)

1. **Вся бизнес-логика — в `server/`**, НЕ в React-компонентах. Компонент отображает и зовёт server-функцию. Логика расчётов, валидаций, движений склада — только в server-слое.
2. **Балансы и стоимость — вычисляются, НЕ хранятся.** Баланс тары = Σ движений. Стоимость = принятый вес × цена строки контракта (живой пересчёт). Никаких закешированных сумм в таблицах.
3. **Единый складской движок.** Тара и ингредиенты — одна сущность StockMovement с дискриминатором `kind`. Не создавай отдельные таблицы движений.
4. **Soft delete** для справочников (active=false), никогда не физическое удаление сущностей со связями.
5. **RBAC на сервере И клиенте.** Скрыть кнопку на фронте недостаточно — server-функция обязана проверять роль (`requireRole`).
6. **ChangeLog** на все правки цены, привязки контракта, статусов, актов приёмки. Запись — через `logChange()` в ТОЙ ЖЕ транзакции, что и изменение. Проверять результат строками в БД, не только отсутствием ошибки.

## Три базы веса (критично, частый источник ошибок)

- **Тара** двигается по **плановому** весу при переходе `planned → sent` (не при создании, не пересчитывается на приёмке).
- **Ингредиенты** расходуются по **фактическому** весу (перевеска).
- **Контракты/стоимость** считаются от **принятого** веса (после брака).
- **Брак** — статистика, база — фактический вес, в выполнение НЕ идёт.

Подробности и все формулы — в DOMAIN.md разделы 1, 4, 5.

## Эталонные паттерны (выработаны на этапе A — копировать, не изобретать)

- **CRUD-модуль:** `server/<entity>/{schema.ts,actions.ts}` + страницы в `app/(app)/...`. Эталон — Farmer (`server/farmers/`).
- **schema.ts** — единая zod-схема (источник типов формы и серверной валидации). Условная валидация — через `superRefine`.
- **actions.ts** — Server Actions, возвращают `ActionResult` (единый формат успех/ошибка). Внутри: `requireRole` → zod-валидация → `$transaction` (запись + `logChange`).
- **Soft delete:** одна server-action `set<Entity>Active(id, active)` туда-обратно, в UI — кнопка по статусу строки (деактивация с AlertDialog).
- **Клиентский RBAC:** `components/auth/RoleGate.tsx`. Серверный: `server/auth/session.ts` → `requireRole`.
- **Валидаторы:** общие в `lib/validators.ts` (`phoneSchema`, `normalizePhone`, `formatPhone` «+7 XXX XXX-XX-XX»). Телефоны в UI — кликабельные `tel:`.
- **FK-Select в формах:** неактивная связанная сущность не теряется при редактировании (добавляется в опции, помечается).
- **Decimal** в Prisma для всех весов/цен/количеств.

## Дизайн (следовать строго)

- Источник: `docs/DESIGN-SYSTEM.md` (токены Vercel), но ТОЛЬКО app-масштаб: radius 6px (кнопки/инпуты/бейджи) и 8px (модалки/контейнеры), form-height 40px (компакт 32px), hairline-бордеры #ebebeb, stacked shadows, Geist; веса и числа — `tabular-nums`; технические метки (№ акта, ISO-недели) — Geist Mono.
- ЗАПРЕЩЕНЫ: mesh-градиенты, pill-кнопки 100px, hero/dark-бенды, эмодзи как иконки. Хром экранов нейтральный светлый — цветовой бюджет отдан статусам и культурам.
- **Статусы отгрузок (две заливки на статус — источник истины: `DESIGN-SYSTEM.md` → `status-fills`).** На статус задаются: `zone` (фон левой зоны машины), `badge` (фон бейджа), `dot` (точка), `fg` (текст). Значения: planned `zone #f5f5f5 / badge #ededed / dot #b3b3b3 / fg #888888` · sent `zone #eaf2ff / badge #d3e5ff / dot #0070f3 / fg #0761d1` · arrived `zone #fff6e3 / badge #ffefcf / dot #f5a623 / fg #ab570a` · accepted `zone #ddfff7 / badge #c7f6ea / dot #29bc9b / fg #1d8e75`. Не использовать `#aaffec` для accepted (это базовый `cyan-soft`, в статусах не применяется).
- **Цвета культур** — из БД (`Culture.color`), НЕ хардкодить. Подложка строки = цвет с opacity 8–10%, чипы = полный цвет.
- Структура экранов — `docs/DESIGN.md`.

## Конвенции кода

- Структура: `app/` (страницы), `server/` (логика, по доменам: shipments/inventory/contracts/acceptance/analytics), `prisma/schema.prisma`, `components/ui/` (shadcn).
- TypeScript строгий. Типы из Prisma переиспользовать, не дублировать вручную.
- Server Actions предпочтительнее ручных REST-эндпойнтов.
- Названия сущностей в коде — как в DOMAIN.md (Shipment, ShipmentItem, StockMovement…).
- Не использовать localStorage/sessionStorage — состояние в React state или БД.

## Принцип работы

- Делаем **вертикальными срезами**: один модуль доводится до рабочего состояния перед следующим. Не распыляться.
- Не over-engineering: без Docker, Redis, микросервисов, GraphQL, очередей. Монолит Next.js, агрегаты на лету.
- Объясняй неочевидные решения коротко — разработчик junior-уровня, код должен быть понятен и сопровождаем.
- После завершённой задачи — обновить статус в TASKS.md и предложить коммит.

## Чего НЕ делать без спроса

- Не менять схему данных (она заморожена) — при необходимости миграции сначала обсудить. **Утверждённая миграция B4** (обсуждена, см. DOMAIN BR-20/21/22): `WeeklyPlan` += `season_year`, `date Date?` (nullable; null=недельная цель) + partial-unique индексы; убедиться, что `ShipmentItem.actual_weight_kg` присутствует в схеме (уже в DOMAIN — добавить колонку, если миграции ещё не было). **Состав недели (B4c, обсуждён, BR-23):** новая таблица `WeeklyPlanScope` (season_year, iso_year, iso_week, culture_id; уникум по неделе+культуре). Иные изменения схемы — по-прежнему через обсуждение.
- Не добавлять зависимости вне утверждённого стека без обоснования.
- Не реализовывать фичи вне текущего этапа TASKS.md (не забегать вперёд по роадмапу).
- Не вставлять код, назначение которого не объяснено.
