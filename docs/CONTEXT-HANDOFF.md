# ARCHITECTURAL DECISIONS (зафиксированы)

**1. Тара двигается ДВУХФАЗНО через транзит-сентинелы.** Локации: `0`=завод, `Farmer.id`, **`-1`=в пути на
завод**, **`-2`=в пути с завода**, `null`=вне системы.
- Причина: пользователю нужны РЕАЛЬНЫЕ остатки и видимость «в пути». Дебет при отправке и кредит при
  прибытии — разные моменты → нужен промежуточный бакет; сентинел-локация сохраняет принцип «баланс = Σ
  движений». Два сентинела (а не один) — чтобы видеть обе дороги раздельно.
- Не менять без обсуждения: число/смысл сентинелов, привязку плеч к статусам.

**2. Начальные остатки — движением `opening`** (`from=null → локация`, source=manual), не отдельной таблицей.
- Причина: единый источник истины (баланс = Σ движений), без второго хранилища и рассинхрона.
- Не менять: не вводить хранимые балансы.

**3. Доставка тары = `MaterialShipment`** (завод→фермер), отдельный рейс, зеркало отгрузки продукции;
**откат обратим в обе стороны** (`arrived→sent` и `sent→planned`).
- Причина: у доставки нет приёмки вниз по цепочке → односторонний `arrived` создавал бы тупик при ошибке.
- Не менять: source движений `material_shipment` (не путать с `shipment`).

**4. Ингредиент расходуется на приёмке** (`consumption`, `from=фермер, to=null`), база = ФАКТИЧЕСКИЙ вес.
- Причина: соль/уксус израсходованы у фермера на продукцию (не едут на завод); BR-4.

**5. Деньги — живой пересчёт, НИЧЕГО не хранится.** Принятый вес — одноступенчато (`computeAcceptedKg`):
`accepted = факт × Σ(принятых %)`; брак и нестандарт — равноправные непринятые доли 100% факта.
- **Два независимых гейта (BR-10):** `is_accepted` → вход в `accepted_weight` (kg-headline);
  `contract_line_id != null` → оплата и выполнение строки. Нестандарт со строкой оплачивается, в
  `accepted_weight` не входит.
- Причина: единый источник истины — инпуты акта; разные вопросы (что принято vs что оплачивается).

**6. Колонки-снимки `accepted_weight_kg` / `brak_weight_kg` — DEPRECATED, считаются на лету.**
- Причина: хранить = второй источник истины и риск рассинхрона. ⚠️ Колонки физически ЕСТЬ в схеме, но НЕ
  пишутся (см. ACTIVE RISKS — противоречие схема↔код).

**7. Единый складской движок `StockMovement` (kind packaging|ingredient).** Баланс = Σ движений по
(локация × тип × состояние good/scrap). Лом — состояние, не локация.

# INVARIANTS
- Всегда вычислять балансы/стоимость/принятый вес (НЕ хранить).
- Никогда не писать `accepted_weight_kg` / `brak_weight_kg` (deprecated).
- Баланс склада = Σ `StockMovement` по (локация × тип × состояние).
- Локации: `0`=завод · `Farmer.id` · `-1`=в пути на завод · `-2`=в пути с завода · `null`=вне системы.
- Движение тары двухфазное: дебет при отправке (→транзит), кредит при прибытии (транзит→получатель).
- Принятый вес — одноступенчато через `computeAcceptedKg`; никакого «годного» = факт×(1−брак).
- Два гейта: `is_accepted` → accepted_weight; `contract_line_id` → оплата/выполнение.
- Ингредиент: расход по ФАКТИЧЕСКОМУ весу.
- Source движений не путать: `shipment` (вход) vs `material_shipment` (доставка) vs `acceptance_act` (ингр.).
- Использовать эталоны: CRUD — Farmer; матрица — NormsMatrix; нетто-сторно/идемпотентность — packaging.ts.
- Логика — в `server/`; компоненты только отображают и зовут server-actions.
- Схему и доки менять только через обсуждение; тонкие вертикальные срезы; context7 для API библиотек.
