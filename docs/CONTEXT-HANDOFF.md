# ARCHITECTURAL DECISIONS
*(решение · причина · что нельзя менять без обсуждения)*

1. **Двухфазное движение тары/материалов через транзит-сентинелы.** Локации: `0`=завод, `Farmer.id`,
   `-1`=в пути на завод (продукция), `-2`=в пути с завода (доставка), `-3`=в пути между фермерами (transfer),
   `null`=вне системы. *Причина:* дебет при отправке и кредит при прибытии — разные моменты → промежуточный
   бакет сохраняет «баланс = Σ движений»; раздельные сентинелы = раздельная видимость дорог. *Не менять:*
   число/смысл сентинелов, привязку плеч к статусам.
2. **Балансы/стоимость/принятый вес — вычисляются, не хранятся** (Σ движений / живой пересчёт). *Причина:*
   единый источник истины, нет рассинхрона. *Не менять:* не вводить кэш-суммы.
3. **Единый `StockMovement` (kind packaging|ingredient); лом — СОСТОЯНИЕ, не локация.** *Причина:* один
   движок, баланс по (локация × предмет × состояние). *Не менять:* не плодить таблицы движений.
4. **Доставка/перенос — один движок, параметризованный по `item_kind` И по origin+транзит.** Делает
   `MaterialShipmentItem` полиморфным (packaging|ingredient, смешанный груз штатно); origin = `source_farmer_id
   ?? завод`, транзит = `-2` (доставка) / `-3` (transfer). *Причина:* не плодить ingredient-/transfer-копии
   плеч (вариант «зеркалить» отвергнут). *Не менять:* единый движок; транзит доставки/ингредиента `-2`,
   transfer `-3`.
5. **Прибытие — по-позиционное (`MaterialShipmentItem.arrived_at`); «частично» — производное.** *Причина:*
   `ShipmentStatus` общий с продукцией → расширять enum нельзя; источник истины прибытия — позиция. *Не менять:*
   отправка whole-trip; «частично» не хранить; откат `sent→planned` только при 0 прибывших.
6. **Гарды идемпотентности ВСЕХ плеч — НЕТТО, не по существованию.** *Причина:* откаты — нетто-сторно
   (append, не delete) → оригиналы остаются; гард-по-существованию блокирует легитимное повторное применение
   после полного цикла (баг materials-fix). *Не менять:* любое новое плечо — нетто-гард.
7. **Деньги — живой пересчёт, два независимых гейта (BR-10):** `is_accepted` → accepted_weight;
   `contract_line_id != null` → оплата/выполнение. Принятый вес одноступенчато (`computeAcceptedKg`).
   *Не менять:* без «годного» = факт×(1−брак); без снимков.
8. **Колонки `accepted_weight_kg`/`brak_weight_kg` — DEPRECATED, не пишутся** (считаются на лету). ⚠️ см. RISKS.
9. **Примитив планирования — одно-фермерская плановая `Shipment`; машины собираются merge'ем.** Фермер на
   `ShipmentItem` (у `Shipment` нет поля фермера; машина = Shipment с позициями нескольких фермеров). Сборка =
   переназначить `shipment_id` позиций + удалить опустевшие; разборка = разбить по `farmer_id`. *Причина:*
   совпадает с реальным процессом (распределить по фермерам/дням → собрать машины); плановые НЕ имеют
   StockMovement (движения при отправке) → merge/disassemble НЕ трогают леджер. *Не менять:* merge/disassemble
   только над planned; не вводить таблицу PlannedPickup (отвергнуто).
10. **Drag-матрица доски:** planned/собранная — пересчёт обеих дат (приб=день колонки, отпр=−2 рабочих дня);
    sent — отправление фиксировано, меняется только прибытие, гард `приб>отпр`; arrived/accepted — locked;
    прошлый день — запрет всем. *Причина:* sent-машина уже уехала (отправление — факт), прибыть раньше
    отправления/в прошлом нельзя. *Не менять* без обсуждения.
11. **Правка отгрузки — через цепочку откатов до planned, не прямой правкой sent/arrived.** *Причина:* тара
    двинута по плановому весу при отправке → правка после отправки десинхронит плечи; откат разматывает
    нетто-сторно. *Не менять:* править позиции только в planned.
12. **Неделя — глобально в URL `?week=YYYY-Www`** (дефолт текущая), общая для /shipments и /planner. *Причина:*
    переживает перезагрузку/шаринг; localStorage запрещён. *Не менять:* состояние недели/вида — только в URL.
13. **Тара и ингредиенты держатся у фермера; дефицит = нужно(норма×план) > баланс фермера.** *Причина:* фермер
    грузит в свою тару и расходует свои ингредиенты; индикатор информационный. *Не менять:* дефицит ничего не блокирует.

# REJECTED ALTERNATIVES
*(предложение · почему отвергнуто · не возвращаться без новых данных)*
- **Зеркальные ingredient-/transfer-копии плеч** (вместо параметризации движка). Дублирование сотен строк
  против «единого движка».
- **`material_shipment_item_id` в леджер** (для дублей позиций). Over-engineering: дубль-строка = одна строка
  с суммой; решено zod-запретом дублей `(farmer,kind,fk)`.
- **Отдельная таблица `MaterialTransfer` / `PlannedPickup`.** Дублирует движок и «план vs факт»; решено
  расширением существующих сущностей.
- **`partially_arrived` в enum `ShipmentStatus`.** Enum общий с продукцией → протечёт; «частично» — производное.
- **Capacity-предупреждение для смешанных машин.** Нет максимума фуры; `TripWeightNorm` — вес рейса
  per фермер×культура. Хинт только для одно-парных карточек.
- **Двухступенчатый расчёт калибра; снимки accepted/brak; оплата нестандарта по умолчанию; один транзит;
  тара одним движением на sent.** (Ранние; заменены текущей моделью.)

# INVARIANTS
- Всегда вычислять балансы/стоимость/принятый вес (НЕ хранить).
- Никогда не писать `accepted_weight_kg` / `brak_weight_kg`.
- Локации: `0`=завод · `Farmer.id` · `-1`=на завод · `-2`=с завода · `-3`=между фермерами · `null`=вне системы.
- Всегда нетто-гард на плечах склада (не по существованию).
- Тара/доставка/transfer — двухфазно (дебет при отправке → транзит; кредит при прибытии → получатель).
- Прибытие — per-item (`arrived_at`); «частично» производное; дубли позиций `(farmer,kind,fk)` запрещены.
- Лом — состояние (scrap), не локация; disposal уводит из системы; ингредиент-`consumption` уводит (`to=null`).
- Принятый вес — одноступенчато (`computeAcceptedKg`); два гейта (is_accepted; contract_line_id).
- Планировать одно-фермерскими; машины — merge'ем; merge/disassemble только над planned, без касания леджера.
- Правка отгрузки — только в planned (через цепочку откатов).
- Неделя/вид — только в URL (`?week=`/`?view=`); НЕ использовать localStorage.
