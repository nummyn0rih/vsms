# CONTEXT HANDOFF — проект VSMS (точка входа в новый чат)

> Актуально на: **этап B закрыт; этап C — C0 (дизайн акта) + C1 (форма акта) закрыты; следующий срез C2**.
> При расхождении с памятью верить актуальным файлам в репозитории/outputs. Модель приёмки и калибров
> финальная (см. «Ключевые доменные решения»).

## РОЛИ И ПРОЦЕСС
Пользователь: снабженец консервного завода, junior frontend. Кодит через **Claude Code (CLI)**,
проектирует через **Claude Design**. Ассистент: контроль процесса, ревью, пишет промпты для Code и
Design, ведёт документацию. Процесс: вертикальные срезы; 1 промпт = 1 проверяемая единица; новый
экран → сперва дизайн-промпт (прототип) → потом код; чеклист ревью после каждого; схему/доки менять
только через обсуждение. Ассистент держит экспертную роль (строкой), критикует конструктивно
(не поддакивает), по-русски, структурно, с блоком «Что делать прямо сейчас». Промпты и обновлённые
доки — в /mnt/user-data/outputs/; брать актуальные файлы оттуда. Один писатель на файл (доки ведёт
ассистент; TASKS — тоже через ассистента, чтобы версии не разъезжались). История этапов в TASKS
задним числом НЕ переписывается (только пометки).

## ЦЕЛЬ
VSMS — внутреннее веб-приложение управления поставками овощного сырья на консервный завод
(замена Excel). Solo + AI. Ядро — **отгрузка (машина/фура)**, приёмка — **позиционная**.
~30 фермеров, 5–10 машин/день, до 5 пользователей (пишет почти всё один админ).

## СТЕК (утверждён, реализован)
Next.js App Router + TS, Prisma, PostgreSQL (**Neon**: pooled DATABASE_URL + direct DIRECT_URL для
миграций), Auth.js v5 (credentials, роль в сессии), Tailwind + shadcn/ui, @dnd-kit, Recharts, SheetJS,
Vercel. Без Docker/Redis/микросервисов/GraphQL/S3. context7 для свежей доки библиотек.

## КЛЮЧЕВЫЕ ДОМЕННЫЕ РЕШЕНИЯ
**Три базы веса:** тара — ПЛАНОВЫЙ (двигается на planned→sent); ингредиенты — ФАКТИЧЕСКИЙ (перевеска,
двигаются на приёмке — C2); контракт/стоимость — ПРИНЯТЫЙ; брак — статистика.
- Сезон июнь→май (BR-17); лето Пн–Сб, зима Пн–Пт (BR-18); ISO-недели; SeasonConfig.workdays.
- **Тара (B2.5):** тип тары — атрибут позиции; культура→типы через m2m (ровно один is_default);
  норма по тройке фермер×культура×тип (нетто); units = ceil(planned/норма); навал → нет движения.
- **Статусы отгрузки:** planned/sent/arrived хранятся; **`accepted` — производный** (все позиции
  приняты). Откат — admin + ChangeLog. planned удаляется физически, sent+ только откат+сторно.
- **Приёмка ПОЗИЦИОННАЯ (BR-13):** единица = ShipmentItem; принята ⇔ есть `AcceptanceAct` (1:1);
  акты позиций — в разное время; статус машины производный («Прибыла»/«Частично принята»/«Принята»).
- **Переходы приёмки (BR-24):** `sent→arrived` авто (первый вес позиции) ИЛИ ручное «Отметить
  прибытие»/шорткат «Акт» из зоны 1 (operator/admin). Односторонний.
- **№ акта (BR-9):** уникален в рамках сезона; хранится `{season_year}-{введённое}` (напр. `2026-9`).
- **Расчёт принятого (BR-10, ОДНОСТУПЕНЧАТО, база=факт):** simple → `accepted = факт×(1−brak%)`;
  calibre → категории схемы + брак делят факт, `Σ=100% от факта`, `accepted = факт×Σ(принятых %)`.
  **brak_percent — поле акта у всех** (НЕ категория); в калибр-форме рендерится последней строкой
  таблицы. Нестандарт (`is_accepted=false`, напр. >12) — «не в зачёт», опц. ручная привязка к строке.
  «Чистый вес»/двухступенчатость НЕ используются (была отвергнута).
- **Калибры:** CalibreScheme/CalibreRange на культуру (редактор в форме Culture при acceptance_type=
  calibre); категории размерные/безразмерные. Привязка категории к строке контракта — на приёмке.
- **Контракты-мини:** Contract + ContractLine (несколько строк/культуру, разные метки/цены, BR-5);
  нестандарт — отдельная строка вручную при оплате. Привязка позиции к строке (BR-7/8).
- **План недели:** WeeklyPlan (день `date` ИЛИ неделя `date=null`, BR-20; конверсия BR-21);
  прогресс по эффективному весу `actual ?? planned` (BR-22, headline только по плановым культурам);
  состав недели WeeklyPlanScope (BR-23, видимость строк = состав ИЛИ цель ИЛИ отгрузки).
- **Склад:** единый StockMovement (kind packaging|ingredient), локации int (0=завод, Farmer.id),
  баланс=Σ движений. Роли: admin / operator (чтение+прибытие+приёмка) / user (чтение).

## UI-СЛОВАРЬ СТАТУСОВ
enum БД: planned/sent/arrived/accepted. Ярлыки ленты: planned→«Плановая» · sent→«Отправлена» ·
arrived→«Прибыла» · accepted→«Принята». Производное «Частично принята · N/M» (часть позиций машины
принята). Бейджи машины в «Приёмке»: «взвешено · N/M» / «принято · N/M». Бейдж принятой позиции —
«Акт принят» (иконка-документ).
