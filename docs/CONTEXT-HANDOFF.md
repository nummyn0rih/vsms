# ARCHITECTURAL DECISIONS
*(зафиксировано; не менять без обсуждения)*

- **Двухуровневый процесс PM↔Claude Code.** Причина: solo-dev с AI; PM держит живую модель и пишет проверяемые
  спеки, Claude Code исполняет в среде с git/тестами. Нельзя менять: PM не редактирует прод-код; фиксы — отдельными
  самодостаточными `PROMPTS-*-FIX.md`; новый экран → дизайн-спека → Claude Design → ревью → код-спека.
- **Балансы/стоимость/принятый вес/выполнение — вычисляются, не хранятся.** Причина: единый источник истины,
  нет рассинхрона. Нельзя: писать закешированные суммы; дублировать формулы агрегации (одна чистая функция на всё).
- **Печать — print-CSS, отдельные bare-роуты `app/print/*`** (НЕ `@media print` поверх интерактивных экранов, НЕ
  серверная PDF-библиотека). Причина: экраны (карточки-леджер, heatmap, Recharts) в чистые A4-таблицы через CSS не
  превращаются; браузерная печать/«Сохранить как PDF» без зависимостей. Нельзя: менять `@media print`-вывод под
  экранные правки; печатать деньги (натуральные единицы). Сводка/План — landscape, остальное portrait.
- **Экспорт Excel — клиентский SheetJS через один хелпер `lib/xlsx-export`.** Причина: фильтры видов клиентские,
  экспорт «что видишь»; одна точка SheetJS. Нельзя: вторая точка использования; числа строками.
- **Палитра культур `CULTURE_PALETTE` (12 цветов) — источник выбора цвета; `Culture.color` остаётся hex.** Причина:
  произвольный hex давал неразличимые/статусные цвета; миграции избежали. Нельзя: ломать off-palette-культуры (их
  цвет — «свой цвет»); запрещать дубли (только предупреждать); хардкодить цвета мимо константы.
- **Аналитика по культуре — отдельный экран `/analytics/culture/[id]`; % брака графиком по неделям; выполнение у
  фермера без контракта = «—».** Причина: профиль просторный и шарабельный; drill-down нужен, чтобы видеть *когда*
  всплеск брака; «—» не путать с «мало принял». Нельзя: деньги-KPI, год-к-году (нет истории).
- **Три базы веса (DOMAIN §1):** тара — плановый (на `sent`); ингредиент — фактический; деньги/принятый — принятый;
  брак — статистика, в выполнение не идёт. Нельзя путать базы.
- **Единый складской движок `StockMovement` (kind), нетто-дисциплина плеч, транзит-сентинелы −1/−2/−3.** Причина:
  один леджер, обратимость, повторный цикл после отката. Нельзя: новое плечо без нетто-гарда; новые сентинелы.
- **Состояние недели/вида/сезона — в URL, не localStorage.** Причина: шарабельность, воспроизводимость.
  (Клиентские фильтры ленты/приёмки — React-Set, сериализуются в URL для печати/экспорта.)

# REJECTED ALTERNATIVES
*(обсуждено и отвергнуто; не возвращаться без новых данных)*

- **Печать акта позиции как первую печатную форму** — предлагалось; отвергнуто: пользователю нужны табличные формы
  рабочих ВИДОВ (Отгрузки/Сводка/План/Приёмка/Аналитика), не документ-акт. `DESIGN-ACT-PRINT.md` помечен замещённым.
- **Упрощать Аналитику до числовых таблиц на печати** — отвергнуто: аналитика с графиками нагляднее (единственный
  печатный вид с графикой). Рабочие данные — таблицами, Аналитика — с графиками.
- **`@media print` поверх интерактивных экранов** — отвергнуто: ledger/heatmap/charts не сводятся к чистым таблицам
  CSS-ом. → отдельные print-представления.
- **Route-group `(print)`** — отвергнуто: не даёт URL-сегмента, `(print)/shipments` → `/shipments` (конфликт). →
  обычный сегмент `app/print/*`.
- **Серверная PDF-библиотека (@react-pdf/pdf-lib)** — отвергнуто в пользу print-CSS (без новых зависимостей, «no
  over-engineering»).
- **% брака одним числом + спарклайн (в аналитике культуры)** — отвергнуто: среднее уже в KPI; график по неделям
  показывает *когда* всплеск. Оставлен bar-по-неделям.
- **Весь апп адаптивно (мобильная версия)** — отвергнуто: часть экранов на телефоне малополезна. Мобиле v1 = 3
  полевых экрана; v2 = read-only Сводка/План.
- **Сравнение сезонов год-к-году сейчас** — отложено (не отвергнуто): нет исторических данных, импорт — этап G.
- **Пер-строковая интенсивность heatmap глобальным максимумом** — отвергнуто: слабые культуры бледнеют → по строке
  культуры.

**Структурные (эпоха склада/движка — единственная запись, в DOMAIN/PRD как «отвергнутое» их нет):**
- **Зеркальные ingredient-/transfer-копии плеч** (вместо параметризации единого движка по `kind`+origin+транзит)
  — дублирование сотен строк.
- **`material_shipment_item_id` в леджер** (ради дублей позиций) — over-engineering; решено zod-запретом дублей
  `(farmer, kind, fk)`, дубль-строка = одна строка с суммой.
- **Отдельные таблицы `MaterialTransfer` / `PlannedPickup`** — дублируют движок и «план vs факт»; решено
  расширением существующих сущностей.
- **`partially_arrived` в enum `ShipmentStatus`** — enum общий с продукцией, протечёт; «частично» — производное
  из `MaterialShipmentItem.arrived_at`.
- **Capacity-предупреждение для смешанных машин** — нет максимума фуры; `TripWeightNorm` — вес рейса
  per фермер×культура; хинт только для одно-парных карточек.
- **Двухступенчатый калибр / снимки accepted-brak / один транзит / тара одним движением на `sent`** — ранние
  модели, заменены текущей (одноступенчатый принятый вес, нетто-сторно, сентинелы −1/−2/−3, двухфазное движение).
