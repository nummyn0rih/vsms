# DESIGN-спека — аналитический дашборд (для Claude Design)

> Задача Claude Design: статичный HTML-прототип дашборда аналитики в стиле проекта (как `lenta-b0.html`,
> `plan-view-b4.html`, `farmer-card-v1.html`). Дальше PM + пользователь ревьюят. Это прототип ВЁРСТКИ и
> ВИЗУАЛЬНОГО ЯЗЫКА графиков — сами графики в проде на **Recharts**; в прототипе рисуй их статично (SVG-макеты
> баров/линий), чтобы задать раскладку, оси, подписи, цвета. Итог — `docs/prototypes/analytics-v1.html`.

## Контекст
VSMS — инструмент снабженца консервного завода. Есть пункт меню «Аналитика» (`/analytics`) — сейчас плейсхолдер.
Нужен дашборд по **текущему сезону**: сводные показатели + графики. Read-only, все роли (аналитика — R).
Данные считаются на лету из отгрузок/приёмок/контрактов. Год-к-году НЕ входит в этот срез (нет истории).

## Раскладка (сверху вниз)
1. **Шапка:** заголовок «Аналитика», селектор сезона (по умолчанию текущий; в проде — URL `?season=`).
2. **KPI-полоса** (4–5 плиток, как в карточке фермера `.kpi`): Принято, т · Выполнение сезона, % ·
   Средний брак, % · Рейсов ТК (овощные+материальные) · Осталось ~N машин.
3. **Сетка графиков** (2×2 или адаптив):
   - **Выполнение по культурам** — горизонтальный bar: по культуре принято (т) на фоне плана (т) + % справа.
     Серия — цвет культуры (из БД).
   - **Динамика приёмки по неделям** — line или area: ось X — ISO-недели сезона, ось Y — тонны (принятый или
     эффективный вес). Возможен разрез по культурам (несколько линий) — но для v1 достаточно суммарной линии.
   - **% брака по культурам** — вертикальный bar: доля брака (actual×brak%/actual) по культурам. Нейтральный
     янтарь для «брака» (как индикатор дефицита B5-2), не алярм-красный.
   - **Рейсы ТК** — вертикальный/горизонтальный bar: число рейсов по транспортным компаниям; овощные и
     материальные — раздельно (BR-14: считаются отдельно). Сгруппированные бары или две серии.

## Состояния (показать)
- Полный сезон с данными — основной вид.
- **Пустой/ранний сезон** (данных почти нет) — аккуратные пустые графики / «данных пока нет», без сломанных осей.
- Одна культура / один ТК — графики не разваливаются.
- (Опц.) переключение сезона — визуально показать селектор.

## Визуальный язык графиков (строго)
- Токены — `docs/DESIGN-SYSTEM.md`. Оси/сетка — hairline `#ebebeb`, подписи — Geist/Geist Mono `tabular-nums`,
  мелкий `--mute`. Никаких тяжёлых сеток, теней, 3D, легенд-простыней.
- **Цвета серий:** культуры — из `Culture.color` (в прототипе — примерные `--cult-*`); нейтральные метрики
  (недели, ТК) — графитовая шкала (`--ink`/серые), НЕ статусные цвета отгрузок. Брак — нейтральный янтарь.
- Плитки/карточки графиков — контейнер `hairline`-бордер + лёгкая тень стека, заголовок графика + краткая
  подпись единиц. Хром нейтральный светлый.
- ЗАПРЕЩЕНО: эмодзи-иконки, алярм-красный, mesh-градиенты, радуга серий, pill-кнопки 100px.

## Границы (реализм)
- Всё read-only, агрегаты вычисляются (не хранятся). Прод — Recharts; прототип — статичные SVG-макеты (числа
  иллюстративные), чтобы утвердить раскладку/оси/цвета до кода.
- Год-к-году, печатные формы, факт. ср. вес рейса как отдельная метрика — НЕ в этом прототипе.
- Селектор сезона в проде — URL (`?season=`), без localStorage.

## Референсы
- KPI-плитки — `farmer-card-v1.html` (`.kpi`), прогресс/бары — `plan-view-b4.html`.
- Статус-заливки/янтарь брака — `DESIGN-SYSTEM.md` (`status-fills`, arrived-янтарь), доска `board-b5-v2.html`.
- Меню/место — `app/(app)/analytics` (плейсхолдер), `lib/nav.ts`.

## Deliverable
`docs/prototypes/analytics-v1.html` — один файл: шапка + KPI-полоса + сетка из 4 графиков (статичные макеты),
плюс пустое состояние. Внизу — блок «Решения/вопросы» (напр. линия суммарная vs по культурам; горизонтальные
vs вертикальные бары; нужен ли разрез брака по фермерам, а не только культурам).

## Ревью v1 (PM + пользователь) — принято, решения зафиксированы
Прототип `analytics-v1.html` утверждён по раскладке. Решения:
- **Динамика приёмки:** v1 — одна суммарная линия/area; разрез по культурам (multi-line + тумблер) — v2.
- **База динамики:** ПРИНЯТЫЙ вес (Σ accepted по неделе прибытия), НЕ эффективный — консистентно с KPI
  «Принято» и словом «приёмка». (Правка относительно подписи прототипа «эффективный».)
- **Выполнение по культурам:** горизонтальные бары (принято/план т + % справа).
- **Брак:** v1 — по культурам (Σ actual×brak% / Σ actual); drill-down по фермерам — v2/карточка фермера.
- **Рейсы ТК:** сгруппированные бары (овощные+материальные раздельно, BR-14), не stacked.
- **«Осталось ~N машин»:** грубая оценка недобор/плановый вес рейса (норма), подпись «≈». Фактический ср.
  вес рейса — отдельная метрика, НЕ в этом срезе.
- Селектор сезона — URL `?season=`, дефолт текущий; опции — из настроенных сезонов (SeasonConfig).
