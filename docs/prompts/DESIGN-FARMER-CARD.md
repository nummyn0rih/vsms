# DESIGN-спека — карточка поставщика (для Claude Design)

> Задача Claude Design: собрать статичный HTML-прототип в стиле проекта (как `lenta-b0.html`,
> `plan-view-b4.html`, `alerts-v1.html`) для экрана карточки фермера. Дальше PM + пользователь ревьюят.
> Это НЕ код приложения — только прототип вёрстки/состояний. Итог — `docs/prototypes/farmer-card-v1.html`.

## Контекст
VSMS — инструмент снабженца консервного завода (замена Excel). Есть справочник фермеров
(`/reference/farmers` — список с фильтром). Нужна **карточка поставщика** — сводный экран по одному фермеру,
куда переходят из списка. Всё **read-only, агрегаты вычисляются на лету** (не хранятся).

## Формат и объём (зафиксировано с пользователем)
- **Отдельная страница** `/reference/farmers/[id]` (не модалка). Переход — из строки списка фермеров.
- **v1 — 4 вкладки:** Основное+Контакты · Контракты · Отгрузки · Тара/ингредиенты.
- **Качество и Аналитика** — показать как **disabled-вкладки «скоро»** (чтобы был виден полный каркас из
  ~6–7 вкладок по PRD Экран 4, но объём v1 читался явно). Не наполнять.
- Активная вкладка — в URL (`?tab=`), без localStorage (конвенция проекта).

## Шапка карточки (над вкладками)
- Имя фермера (крупно) + бейдж статуса (активен / архивный — как в справочниках).
- Быстрые контакты: телефон кликабельный (`tel:`, формат «+7 XXX XXX-XX-XX»), контактное лицо, мессенджер/почта
  если есть. Кнопка «К списку».
- Мини-сводка (2–4 показателя): напр. активных контрактов N · выполнение сезона % · тара на балансе · сезон.
  (Числа иллюстративные.)

## Вкладки v1 — содержимое
1. **Основное + Контакты.** Заметки (notes), структурированные контакты (телефон tel:, контактное лицо,
   мессенджер, email), статус active, сезон. Спокойная форма-просмотр (не редактирование — правка в справочнике).
2. **Контракты.** Список контрактов/строк фермера с ЖИВЫМ выполнением и стоимостью (источник —
   `getContractExecution`, C3a/C3d): по строке — культура, объём план (т), принято (т), выполнение %,
   цена ₽/кг, стоимость ₽. Итоги по контракту/фермеру. Стиль — как блок выполнения в `ContractViewDialog`.
3. **Отгрузки (история).** Список отгрузок фермера (через `ShipmentItem`), сгруппированный по неделям/датам:
   дата, № машины, статус (бейджи статусов из ленты — planned/sent/arrived/accepted), культура, план/факт/
   принято (кг). Стиль — компактные строки как в «Ленте» (`lenta-b0`), но плоско по фермеру.
4. **Тара / ингредиенты.** Балансы У ЭТОГО фермера (источник — `getTareBalances`/`getIngredientBalances`,
   фильтр по локации-фермеру): тара по типам × состояние (good/лом), ингредиенты по позициям с ед. (кг/л);
   строки транзита, если едет. Стиль — как мини-матрица дашбордов `/packaging`, `/ingredients`.

## Состояния (показать в прототипе)
- Фермер с полными данными (все 4 вкладки наполнены) — основной вид.
- Пустые вкладки: нет контрактов / нет отгрузок / нулевые балансы — аккуратные пустые состояния.
- Архивный фермер (active=false) — бейдж, но карточка доступна (история сохраняется).
- Длинные списки отгрузок — поведение (скролл/группировка по неделям).
- Пример активной вкладки и disabled «скоро» (Качество/Аналитика).

## Визуальный язык (строго)
- Токены — `docs/DESIGN-SYSTEM.md`: radius 6/8px, hairline `#ebebeb`, Geist; числа/веса `tabular-nums`,
  техметки (№ акта, ISO-недели) — Geist Mono. Статусы отгрузок — канонические заливки из DESIGN-SYSTEM
  (planned/sent/arrived/accepted). Цвета культур — из БД (чипы полным цветом, подложка ~9%).
- Вкладки — shadcn Tabs-стиль (сегмент), как в проекте. Хром нейтральный светлый.
- ЗАПРЕЩЕНО: эмодзи как иконки, алярм-красный вне статусов, pill-кнопки 100px, mesh-градиенты, hero-бенды.

## Границы (реализм)
- Всё read-only. Правка данных фермера — в справочнике, не в карточке (в карточке только просмотр + `tel:`).
- Выполнение/стоимость/принятый вес/балансы — вычисляемые (не хранить). Брак/аналитика — вкладки-заглушки v1.
- Опционально (idea, не обязательно): из строки отгрузки — ссылка в ленту; из строки контракта — в контракт.

## Референсы для консистентности
- `docs/prototypes/lenta-b0.html` — строки отгрузок, бейджи статусов, шапки недель.
- Блок выполнения контракта — экран `ContractViewDialog` (код `app/(app)/contracts`).
- Дашборды балансов — `app/(app)/packaging`, `app/(app)/ingredients`.
- Справочник фермеров (шапка/статус/контакты) — `app/(app)/reference/farmers`, `lib/validators` (formatPhone).

## Ревью v1 (PM + пользователь) — правки к ревизии
Прототип принят по структуре. Две правки обязательны + ответы на открытые вопросы.

**Обязательные фиксы:**
1. **Артефакт Cloudflare email-protection.** Email отрисован как `[email protected]` через
   `__cf_email__`/`/cdn-cgi/l/email-protection` + внедрён скрипт `email-decode.min.js`. Это артефакт
   пайплайна, не дизайн. Вернуть обычный email в разметке, убрать cf-спаны/href и скрипт.
2. **Выравнивание колонок истории отгрузок (C2).** Сейчас каждая неделя — отдельная `<table>` с
   `table-layout: auto` → колонки недель не совпадают по ширине. Задать общие фиксированные ширины:
   `table-layout: fixed` + единый `<colgroup>` (одинаковые ширины на всех `.ship-tbl`), либо одна таблица
   с неделей-разделителем. Итог: колонки Дата/Машина/Статус/Культура/План/Факт/Принято стоят друг над другом.

**Ответы на «Решения/вопросы» прототипа (зафиксировано):**
- **Группировка отгрузок:** v1 — плоские строки-позиции (машина повторяется). Сворачивание позиций одной
  машины — бэклог (не в v1).
- **Глубина истории:** v1 — текущий сезон; переключатель сезона — бэклог. Для архивного фермера достаточно
  «последней отгрузки» в мини-сводке.
- **«Принято» до приёмки:** оставить `— приёмка` (sent/arrived) / `—` (planned). Ок.
- **Выполнение в шапке vs итог по контрактам:** ОДИН источник (`getContractExecution`) и одно округление —
  расхождения 64/65% быть не должно. Шапочный KPI считать из того же агрегата, что и итог вкладки.
- **Тара «в пути»:** показывать транзит консистентно с дашбордом `/packaging` (те же направления/сентинелы),
  без изобретения своей логики; обратный транзит (фермер→завод) — как на дашборде.
