# _PM-WORKFLOW — как работает связка PM-ассистент ↔ Claude Code

> Операционная инструкция по двухуровневой разработке VSMS. Один живой PM-контекст + руки в репозитории.
> Не заводить отдельных агентов под обсуждения/архитектуру/объяснения — это PM в роли-хате.

## Два уровня
- **PM-ассистент (этот контекст):** анализ архитектуры, спеки, промпты для Claude Code, доки/handoff,
  долгосрочная память, брейншторм, проектирование решений. Держит ЖИВУЮ модель проекта. Не редактирует код.
- **Claude Code (CLI в WSL):** реализация — правка кода, тесты, git, терминал, миграции. Стартует с репо +
  CLAUDE.md + приложенной спекой.

## Контур передачи (один цикл задачи)
1. PM: анализ → спека-промпт в `docs/prompts/PROMPTS-<TASK>.md` (шаблон ниже).
2. Claude Code: реализует строго по спеке, гоняет проверки, предлагает коммит.
3. (опц.) Claude Code: прогон repo-агента `invariant-review` перед коммитом.
4. PM: обновляет `TASKS.md` + `CONTEXT-HANDOFF.md` (и `DOMAIN.md`, если менялись правила).
5. (фон) scheduled-гигиена раз в неделю ловит расхождения доки↔код.

## Активы PM-слоя (минимальный набор, не разрастаться)
- **Шаблон спеки** — ниже. Каждый промпт = 1 проверяемая единица.
- **On-demand аудитор согласованности** — read-only субагент, спавнится ПО ЗАПРОСУ (промпт ниже). Изолированный
  fan-out по файлам, возвращает только расхождения, не грузит PM-контекст.
- **Scheduled-гигиена** — `.claude/Scheduled` задача `vsms-memory-hygiene` (еженедельно). Сверяет
  CONTEXT-HANDOFF/TASKS с `.git/logs/HEAD` и кодом, отдаёт короткий дайджест дрейфа.
- **Repo-агенты Claude Code** — `.claude/agents/invariant-review.md`, `.claude/agents/test-runner.md`
  (живут в репо, доступ к git/терминалу).

## ШАБЛОН спеки PM→Claude Code
Скопировать в `docs/prompts/PROMPTS-<TASK>.md`, заполнить, отдать Claude Code.

```text
# PROMPTS — <task-id>: <одна строка цели>

> <2–3 строки: какой пробел закрываем и почему. Есть ли миграция (да/нет).>

## Решения (зафиксированы)
<таблица «развилка → выбранное поведение», если были продуктовые вопросы>

## ПРОМПТ — <task-id> (Claude Code)
```text
Задача <task-id> (VSMS): <цель в 1–2 предложения>. Миграция: <нет / какая>.
Перед кодом прочитать: <точные файлы server/* и app/*> + DOMAIN.md <§/BR>. Next/Prisma/Auth.js — context7.

СЕРВЕР
1) <файл> — <что изменить, какие функции, инварианты: нетто-гард, $transaction+logChange, requireRole>
...

UI
N) <файл/компонент> — <что показать/звать; RBAC клиент>

ОГРАНИЧЕНИЯ
- <схема/миграция; не хранить балансы; не писать accepted_weight_kg; RBAC сервер+клиент; доки не трогать (PM)>

БД ТЕСТОВАЯ: проверки seed/rolled-back.

ПРОВЕРКА (показать)
- <сценарий 1 ... ожидаемый результат>
- <RBAC: non-роль отклонена>
В конце предложи git-коммит одной строкой.
```

### Чеклист ревью <task-id>
1. <серверный инвариант>
2. <идемпотентность/нетто-гард>
3. <UI/RBAC>
4. <без миграции / схема цела>
```

**Правила хорошей спеки (держать в голове):**
- 1 промпт = 1 проверяемая единица. Большое — резать на срезы.
- Всегда: какие файлы прочитать ПЕРЕД кодом; какие BR/§ DOMAIN затронуты; есть ли миграция.
- Всегда явные ОГРАНИЧЕНИЯ (нетто-гард, logChange в транзакции, RBAC сервер+клиент, «не писать deprecated-колонки»).
- Всегда блок ПРОВЕРКА с конкретными сценариями + RBAC-негатив. БД тестовая → rolled-back tx.
- Доки/TASKS Claude Code не трогает — это зона PM.
- context7 для API Next.js/Prisma/Auth.js/shadcn/@dnd-kit/Recharts/SheetJS.

## ПРОМПТ on-demand аудитора согласованности (read-only)
Спавнить субагентом ПО ЗАПРОСУ пользователя («сверь доки с кодом»). Только чтение, без правок.

```text
Read-only аудит согласованности VSMS. НИЧЕГО не менять, только читать и вернуть отчёт.
Источники: docs/CONTEXT-HANDOFF.md, docs/TASKS.md, docs/DOMAIN.md, prisma/schema.prisma, server/**, app/**,
.git/logs/HEAD (текстовый reflog — последние коммиты).

Проверить и вернуть СПИСКОМ только РАСХОЖДЕНИЯ (без пересказа того, что совпадает):
1. CONTEXT-HANDOFF ↔ код: заявленные «закрыто/работает» срезы реально присутствуют в server/app (по ключевым
   функциям/файлам). Что заявлено, но не найдено — и наоборот (код есть, в handoff не отражён).
2. CONTEXT-HANDOFF ↔ TASKS: статусы задач согласованы (нет «[ ]» на то, что handoff считает закрытым, и наоборот).
3. CONTEXT-HANDOFF ↔ DOMAIN: BR-номера и инварианты, на которые ссылается handoff, существуют в DOMAIN и не
   противоречат ему.
4. Дрейф handoff относительно репо: последние коммиты в .git/logs/HEAD, не отражённые в SESSION SUMMARY/PROJECT STATE.
5. Нарушения инвариантов в коде (быстрый скан, если попадётся): запись accepted_weight_kg/brak_weight_kg; новое
   плечо склада без нетто-гарда; server-action без requireRole; изменение без logChange в транзакции.

Формат отчёта: для каждого расхождения — [где] · [в чём] · [вероятная причина] · [серьёзность low/med/high].
Не делать предположений без опоры на файл/строку. Не предлагать правок — только находки.
```

## Что НЕ заводим (осознанно)
- Отдельных агентов под брейншторм / объяснения / архитектуру / проектирование — теряется общий контекст,
  ради которого держим единый PM-слой. Это PM в роли-хате (§3 преференсов).
- Больше 3 активов на стороне PM и 2–3 repo-агентов — agent sprawl для solo-dev не окупается.
