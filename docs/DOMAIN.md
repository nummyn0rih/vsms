## 1. Три базы расчёта веса (КРИТИЧНО — ядро всех расчётов)

Самая частая ошибка — перепутать базу. Каждый расчёт использует строго определённый вес:

| Что считаем | База веса | Почему |
|---|---|---|
| Тара (движение) | **Плановый** (planned_weight) | Отправили 1000 ящиков — пришла 1000, вне зависимости от приёмки |
| Ингредиенты (расход) | **Фактический** (actual_weight, перевеска) | Заложены в производство ДО браковки |
| Контракт + стоимость | **Принятый** (accepted_weight, после брака) | Платим только за принятое |
| Брак (статистика) | **Фактический** (actual_weight, знаменатель) | % брака = непринято / перевеска |

**Три веса позиции (ShipmentItem):**
- `planned_weight_kg` — приблизительный, при создании отгрузки.
- `actual_weight_kg` — после сплошной перевески на воротах.
- `accepted_weight_kg` — производное (одноступенчато, база = факт): **simple** = actual × (1 − brak%); **calibre** = actual × Σ(% принятых категорий). Брак — поле акта у всех (у calibre брак входит в 100% факта наравне с категориями). **Вычисляется на лету (`server/acceptance/accepted.ts` → `computeAcceptedKg`); одноимённая колонка БД — снимок-заглушка, фактически НЕ пишется (deprecated, см. §5). Источник истины — инпуты акта (actual, brak%, CalibreResult.percent).**

## 3. Единый складской движок (StockMovement)

Тара и ингредиенты — ОДИН движок, дискриминатор `kind`.

`StockMovement`: id, date, kind {packaging\|ingredient}, item_id, quantity, from_location_id, to_location_id, from_state, to_state, movement_type, source_doc_type, source_doc_id.

`movement_type`: return (тара фермер→завод с продукцией, **двухфазно через транзит**, из Shipment) · delivery (тара завод→фермер, **двухфазно через транзит**, из MaterialShipment) · transfer (фермер→фермер) · scrap (годная→лом, same loc) · disposal (лом→out, утиль) · adjustment (ручная правка) · **consumption (расход ингредиента на приёмке: from=фермер → to=null, source=acceptance_act; база = факт × IngredientRecipe; УВОДИТ из системы)** · **opening (начальный остаток: from=null→локация, source=manual; задаётся один раз, редактируется заменой)**.

**Правила движка:**
1. **Локация** = завод (фикс. id=0) ∪ Farmer.id ∪ **транзит-сентинелы: `-1` = в пути НА завод (тара с продукцией от фермера), `-2` = в пути С завода (тара к фермеру)**. `null` = вне системы. Лом/утиль — НЕ локации (это состояние). Константы — в `server/shipments/packaging.ts` (`FACTORY_LOCATION_ID=0`, `TRANSIT_TO_FACTORY=-1`, `TRANSIT_TO_FARMER=-2`).
2. **Лом — это состояние** (`state`), не место. «Списать в лом» = движение годная→лом на той же локации. «Утилизация» = disposal лома из системы. Баланс лома по фермеру виден (для сбора целой машиной).
3. **Движение тары — ДВУХФАЗНОЕ, привязано к статусам** (дебет при отправке, кредит при прибытии; между — транзит). Количество тары = ceil(planned_weight / PackagingNorm.avg_unit_weight по тройке фермер×культура×тип); навал (тип=null) → движения нет; нет нормы → переход заблокирован. capacity_kg не участвует.
   - **Входящая (Shipment, продукция фермер→завод):** `planned→sent` → `фермер → -1`; `sent→arrived` → `-1 → завод`. Откат статуса (Admin) сторнирует своё плечо. Оба плеча — одинаковое количество (по плановому весу), не пересчитывается на приёмке.
   - **Доставка (MaterialShipment, тара завод→фермер):** отдельный рейс (свой водитель/даты, груз=тара, кол-во задаётся в MaterialShipmentItem.quantity, не из веса). `planned→sent` → `завод → -2`; `sent→arrived` → `-2 → фермер`. Откат сторнирует плечо. С входящим рейсом НЕ связан. Груз рейса — тара И/ИЛИ ингредиент (`item_kind`); **смешанный груз — штатно**, плечи строятся построчно по `kind`+FK (единый движок, `server/materials/movements.ts`). Ингредиент использует ТОТ ЖЕ транзит `-2`; на завод (`-1`) ингредиент не ходит.
   - **Начальный остаток (opening):** разовое движение `from=null → локация` (завод и фермеры по типам), `movement_type=opening`, `source=manual`. Редактирование — заменой opening-движения (это setup-данные, не операционный леджер). Для ингредиентов opening аналогичен (локации завод/фермеры × ингредиенты), кол-во — Decimal (кг/л).
4. **Авто-расход ингредиентов (`consumption`)** на приёмке: `actual_weight × IngredientRecipe` (6 знаков), движение `from=фермер → to=null`, `source=acceptance_act`. Расход уводит количество из системы (не едет на завод). База — ФАКТ (BR-4).
5. **Баланс**(location, item, state) = Σ in − Σ out. Может быть ОТРИЦАТЕЛЬНЫМ (разрешено).

**Ручные операции тары (D2-ops, source=manual, admin):** лом — это СОСТОЯНИЕ, не локация.
- `scrap` (списание в лом): `loc/good → loc/scrap`, та же локация (тара не уезжает, меняется состояние).
- `disposal` (утиль): `loc/scrap → null` (выводит лом из системы).
- `adjustment` (корректировка как инвентаризация): оператор вводит ФАКТИЧЕСКИЙ остаток ячейки
  (локация × тип × состояние); сервер считает текущий Σ и пишет ОДНО движение adjustment на разницу
  (`delta>0`: `null→loc/state`; `delta<0`: `loc/state→null`; `delta=0`: без движения).
Локации операций — завод/фермер (вкл. архивного); транзит(-1/-2)/null запрещены. Кол-во тары — целое.
Жёсткого потолка нет (баланс может быть отрицательным). Причина операции — в ChangeLog (у StockMovement
поля нет). Точка входа — drill-down диалог ячейки на /packaging (good- и scrap-вид кликабельны).
6. Дефицит НЕ блокирует отгрузку — только информационные алерты (AlertRule.threshold).

## 4. Бизнес-правила (инварианты)

| # | Правило |
|---|---|
| BR-1 | Выполнение контракта и стоимость считаются ТОЛЬКО от принятого веса. Брак не уменьшает и не участвует. |
| BR-2 | Брак — чисто статистический параметр (% по фермеру/культуре/сезону), база — фактический вес. |
| BR-3 | Тара двигается ДВУХФАЗНО (по плановому весу), привязано к статусам: продукция `sent` списывает у фермера (→ в пути на завод `-1`), `arrived` зачисляет на завод (`-1` →завод); доставка тары `sent` списывает с завода (→ в пути с завода `-2`), `arrived` зачисляет фермеру (`-2`→фермер). На `planned` — информационно, баланс не меняется. Откат статуса (Admin) сторнирует соответствующее плечо обратным `StockMovement`. НЕ пересчитывается на приёмке. |
| BR-4 | Ингредиенты расходуются по фактическому весу (перевеска), т.к. заложены до браковки. |
| BR-5 | Цена — это строки контракта (ContractLine). Одна культура может иметь несколько строк с разной ценой. |
| BR-6 | Стоимость/выполнение — ЖИВОЙ пересчёт: при изменении цены строки или перепривязки все привязанные позиции пересчитываются автоматически (цифры не хранятся). |
| BR-7 | Перепривязать ShipmentItem можно только к строке ТОЙ ЖЕ культуры и ТОГО ЖЕ фермера. Блокирующая валидация. То же для привязки категории приёмки (CalibreResult.contract_line_id) — только строка той же культуры и того же фермера. |
| BR-8 | contract_line_id обязателен (NOT NULL) при переходе позиции в accepted. Если строк культуры >1 — выбор обязателен; если 1 — авто. |
| BR-9 | `act_number` = номер партии, присваивается на заводе, **обязателен при accepted**. Уникален **в рамках сезона** (season_year), не глобально. Ввод — свободный (цифры/буквы/символы); при хранении префиксуется годом сезона: хранится как `{season_year}-{введённое}` (напр. `2026-9`). Уникальность проверяется по паре (season_year, act_number-введённое) на сохранении. Каждая позиция машины = своя партия = свой акт. |
| BR-10 | `brak_percent` (порча, % от actual) — поле акта у **ВСЕХ** культур (simple и calibre), НЕ категория. **Одноступенчато, единая база = факт (actual):** `Σ(% размерных/безразмерных категорий схемы) + brak% = 100% от actual`. `accepted = actual × Σ(% категорий is_accepted=true)`. Эквивалентно `actual × (1 − brak% − Σ нестандарт%)`. **simple:** размерных категорий нет → `accepted = actual × (1 − brak%)`. **calibre:** категории схемы (размерные/безразмерные, напр. 6–9, 9–12, >12, или «Стандарт»/«Пульпа») + брак замыкают 100% факта. Неприёмные категории (`is_accepted=false`, напр. >12) и брак — не в зачёт. **В UI калибр-формы `brak_percent` отображается последней строкой таблицы категорий** (нередактируемая «Строка контракта» = «— не в зачёт»); хранится как поле акта, не как `CalibreResult`. Дефолт привязки: `is_accepted=true` → наследует `ShipmentItem.contract_line_id`; `is_accepted=false` → null (нестандарт без оплаты; админ может вручную привязать к строке). **Два НЕЗАВИСИМЫХ гейта (не путать): `is_accepted` определяет вход в `accepted_weight` (стандартный «к оплате», kg-headline); `contract_line_id != null` определяет ОПЛАТУ и выполнение строки. Нестандарт со строкой оплачивается на ней (свой объём, своя цена), но в `accepted_weight` НЕ входит. Поэтому стоимость/выполнение считаются «по категориям с привязкой к строке», а `accepted_weight` — «Σ принятых». Пока нестандарт без строки (line=null) — обе суммы совпадают.** «Чистый вес»/двухступенчатость не используются. |
| BR-11 | Дата прибытия не должна падать на нерабочий день завода (валидация по SeasonConfig). |
| BR-12 | При создании отгрузки вводится дата прибытия ИЛИ отправления, вторая = ±2 дня. Обе правятся вручную потом. |
| BR-13 | **Приёмка ПОЗИЦИОННАЯ:** единица приёмки = ShipmentItem (BR-9, «каждая позиция = партия»). Позиция принята ⇔ существует её `AcceptanceAct` (1:1). Акты позиций одной машины оформляются в РАЗНОЕ время (частая ситуация). **Статус машины:** хранятся `planned/sent/arrived` (BR-11/12/24); `accepted` — АВТО, выставляется когда ВСЕ позиции машины приняты; руками не ставится; при откате акта любой позиции (admin) машина `accepted → arrived`. UI-состояние **«Частично принята»** (≥1 позиция принята, но не все) — ПРОИЗВОДНОЕ, не хранится (хранимый статус остаётся `arrived`). Лента/Приёмка показывают производный статус. Откат — только Admin, в ChangeLog. |
| BR-14 | Рейс ТК = факт прибытия/доставки машины. Овощные и материальные рейсы считаются раздельно. Фермер→фермер и лом/утиль рейсами не считаются. |
| BR-15 | Удаление справочников со связями — только soft delete (active=false). |
| BR-16 | Все правки цены/привязки/статусов/приёмки логируются в ChangeLog. |
| BR-17 | Номера недель — ISO-календарные. season_year = год начала сезона (июнь). |
| BR-18 | Рабочие дни: лето (июнь–сент) Пн–Сб (6), зима (окт–май) Пн–Пт (5). Берутся из SeasonConfig. |
| BR-19 | Отгрузка в статусе `planned` — черновик без движений, может быть удалена физически. Со статуса `sent` и далее удаление запрещено: сначала откат статуса Admin (со сторно движений по BR-3), затем удаление. |
| BR-20 | План недели (WeeklyPlan) задаётся для пары (неделя, культура) в ОДНОЙ гранулярности: либо дневные строки (`date` задан; можно не на все дни), либо одна недельная (`date=null`). Смешивать в рамках одной (неделя, культура) запрещено — иначе двойной учёт. Контроль: server-валидация при сохранении + partial-unique индексы. Прогресс при дневной гранулярности раскладывается по дате прибытия; при недельной — суммируется на неделю. |
| BR-21 | Конверсия гранулярности (атомарно, $transaction, в ChangeLog), итог сохраняется: **день→неделя** — дневные строки (неделя,культура) заменяются одной недельной, `target_tons = Σ дневных`. **неделя→дни** — недельная строка заменяется дневными по рабочим дням этой ISO-недели (SeasonConfig); target распределяется равномерно, остаток — на последний рабочий день (Σ точно сохраняется); далее правится вручную. |
| BR-22 | **Прогресс плана считается по ЭФФЕКТИВНОМУ весу** позиции: `effective = actual_weight_kg ?? planned_weight_kg` (есть перевеска → по ней, иначе по плановому). Это эвристика прогресса планирования, НЕ одна из трёх учётных баз (§1) и НЕ «принятый». В прогресс идут все неудалённые отгрузки (planned/sent/arrived/accepted). Дефицит/перелив информативны, не блокируют. **Headline «выполнение плана» (набрано/цель)** считается ТОЛЬКО по культурам, у которых есть цель: Σ effective(культуры с целью) / Σ целей. Факт культур без цели в headline НЕ идёт (остаётся в их строках). |
| BR-23 | Видимость строки культуры в виде «План» недели = (культура в `WeeklyPlanScope` этой недели) ИЛИ (есть цель `WeeklyPlan` на неделе) ИЛИ (есть неудалённые отгрузки культуры на неделе). `WeeklyPlanScope` управляет только ПУСТЫМИ культурами (заранее открыть под план). Удаление культуры из scope не трогает её цели/отгрузки; если они есть — строка остаётся видимой по другим условиям. **week-mode культура** (цель `date=null`): её цель и факт идут ТОЛЬКО в колонку «Неделя»; в дневных колонках и в дневных подытогах `tfoot` она НЕ участвует (дневной подвал = только day-mode культуры; колонка «Неделя» подвала = все культуры). Редактирование состава — admin; видят все. |
| BR-24 | Переход `sent → arrived` — двумя путями, оба operator/admin: **(а) авто** при сохранении первого `actual_weight_kg` любой позиции машины; **(б) явным действием «Отметить прибытие»** без веса (для машин, чьё сырьё не перевешивают на воротах, — иначе машина вечно висит `sent`). Переход **односторонний**: откат `arrived → sent` — только admin вручную (со сторно по BR-3/19). `arrived` НЕ требует наличия весов (норма для «только патиссоны»). |
| BR-25 | `actual_weight_kg` (перевеска) обязателен для перевода позиции в `accepted`: приёмка позиции невозможна без фактического веса. Вес вносится в ЛЮБОЙ момент — заранее инлайн в «Приёмке» (зона перевески) ИЛИ в форме акта (тогда подставляется/вводится там же). Инлайн-ввод и форма акта пишут ОДНО поле `actual_weight_kg` (один источник, два входа). % брака считается ОТ `actual_weight_kg` (BR-5). Инвариант проверяется на сохранении приёмки (не блокировкой поля). |
| BR-26 | Зоны вкладки «Приёмка» (производные, не хранятся): **зона 1 «Ожидают перевески»** = машины `sent`; **зона 2 «На приёмке»** = машины `arrived`, у которых НЕ все позиции приняты (включая «ни одной»; **частично принятые — здесь**, до приёмки последней позиции); **зона 3 «Принято»** = машины со ВСЕМИ принятыми позициями. Приёмка идёт ПО ПОЗИЦИЯМ (кнопка «Акт» — на позиции `arrived+`, не на машине); машина перетекает 2→3 авто по BR-13. Статус руками не меняют (кроме отката admin). |
| BR-27 | **Баланс ингредиента** = Σ движений по (локация × ингредиент); состояние всегда `good` (у ингредиента нет лома). Транзит — только `-2` (доставка). **«Итого в системе» по ингредиенту = Σ opening + Σ доставлено(нетто 0) − Σ consumption** → расход штатно уменьшает итог; инвариант «= Σ opening» (как у тары) для ингредиентов НЕ действует — это норма, не утечка. Единицы (кг/л) НЕ суммируются между собой (каждый ингредиент — своя единица). Отрицательный баланс фермера допустим (расход раньше прибытия доставки). |
| BR-28 | **Ручные операции тары** (admin, source=manual): scrap (`loc/good→loc/scrap`), disposal (`loc/scrap→null`), adjustment-инвентаризация (ввод факта → движение на разницу `delta`). Лом — состояние, не локация. Транзит/null — не локации операций; локация фермера допустима даже архивная (лом мог застрять). Потолка по количеству нет (баланс может быть отрицательным). Причина — в ChangeLog. Tare-only. |

## 5. Производные величины (вычислять, НЕ хранить)

> **Колонки-снимки `ShipmentItem.accepted_weight_kg` и `AcceptanceAct.brak_weight_kg` — DEPRECATED:**
> фактически НЕ пишутся; принятый/браковый вес выводятся на лету из инпутов акта
> (`computeAcceptedKg`, `accepted.ts`). Снимок создавал бы второй источник истины и риск
> рассинхрона при правке акта — противоречит принципу «вычислять, не хранить» (CLAUDE.md §2).
> Снос колонок — отдельной cleanup-миграцией (этап D), не блокирует C3.
> _Заметка (на будущее, не C3):_ `is_accepted` берётся из текущей схемы калибров (`CalibreRange`),
> а не из снимка акта — при правке схемы исторические accepted пересчитаются. Решить
> «морозить is_accepted в CalibreResult vs живо», когда появится workflow редактирования схем.

| Что | Формула | База |
|---|---|---|
| accepted_weight | simple: actual×(1−brak%) · calibre: actual×Σ(принятых категорий %) | принятый |
| % выполнения строки | Σ accepted (farmer×culture×line) / volume_tons | принятый |
| стоимость | Σ (accepted × price_per_kg строки) | принятый |
| % брака | Σ (actual × brak%) / Σ actual | фактический |
| распределение калибров | Σ (actual × калибр%) / Σ accepted | фактический→принятый |
| баланс тары/ингр. | Σ движений (по location, item, state) | — |
| рейсы ТК (овощи) | COUNT Shipment по ТК в arrived/accepted | — |
| рейсы ТК (материалы) | COUNT MaterialShipment доставленных | — |
| осталось ~N машин | (volume − accepted) / TripWeightNorm | принятый |
| прогресс плана (день/неделя) | Σ effective веса позиций / WeeklyPlan.target_tons (по оси date×culture или iso_week×culture; кг→т) | **эффективный = actual ?? planned** (BR-22) |
| недельные/дневные итоги | агрегаты ShipmentItem по arrival_date | — |

**Для calibre-культур** (одноступенчато: все категории + брак делят факт, Σ=100% от actual; каждая категория со своей привязкой к строке):

| Что | Формула | База |
|---|---|---|
| выполнение строки | Σ (actual × % категорий, привязанных к ЭТОЙ строке) / volume_tons | принятый |
| стоимость | Σ по категориям (actual × % × price_per_kg ИХ строки) | принятый |
| accepted_weight позиции | actual × Σ(% принятых категорий) | принятый |
| нестандарт | Σ (actual × % категорий is_accepted=false с размерами) | принятый-сорт |
| брак (порча) | actual × brak% | фактический |

- **Σ = 100% от факта:** `Σ(% категорий схемы) + brak% = 100%`. Брак — поле акта; в калибр-форме рендерится последней строкой таблицы категорий. «Чистый вес»/двухступенчатость не используются.
- **Нестандарт** (категории `is_accepted=false` с размерами, напр. >12) — НЕ брак (годен по сохранности, неприёмен по размеру); отдельная статистика, в accepted не входит.
- **Объёмы стандарта и нестандарта не смешиваются.** Даже если нестандарт оплачивается по цене стандарта — это ОТДЕЛЬНАЯ строка контракта (та же цена, свой объём). Категории привязаны к разным строкам → их объём идёт в выполнение разных строк.

## 6. Справочные константы домена

- Типы тары: ящик овощной (томаты, черри); бочка металл 200 кг (патиссоны маринованные); бочка пластик 250 кг (патиссоны, перец сладкий, халапеньо — маринованные).
- Ингредиенты: соль, уксус, аскорбиновая кислота, пиросульфит (для маринованных).
- Калибруемая культура (примеры): огурцы — `brak_percent` (порча) + **размерные** категории годного: 6–9 см (принято), 9–12 см (принято), >12 см (нестандарт, не принято). Томаты как calibre — `brak_percent` + **безразмерные** категории: «Стандарт» (принято), «Пульпа» (принято/нет по договорённости). Брак — поле акта у любой культуры, НЕ категория.
- Сезон: июнь→май. season_year = год июня.
