# ARCHITECTURAL DECISIONS (зафиксированы)

**1. Тара/материалы двигаются ДВУХФАЗНО через транзит-сентинелы.** Локации: `0`=завод, `Farmer.id`,
**`-1`=в пути на завод**, **`-2`=в пути с завода**, `null`=вне системы.
- Дебет при отправке и кредит при прибытии — разные моменты → промежуточный бакет; сентинел-локация
  сохраняет «баланс = Σ движений». Два сентинела — раздельная видимость двух дорог.
- Не менять: число/смысл сентинелов, привязку плеч к статусам.

**2. Начальные остатки — движением `opening`** (`from=null → локация`, source=manual), не отдельной таблицей.
Для ингредиентов аналогично (кол-во — Decimal кг/л). Редактирование — заменой движения.

**3. Доставка = `MaterialShipment`** (завод→фермер), откат обратим в обе стороны (`arrived→sent`, `sent→planned`).
Source движений `material_shipment` (не путать с `shipment`).

**4. Ингредиент расходуется на приёмке** (`consumption`, `from=фермер, to=null`), база = ФАКТИЧЕСКИЙ вес
(BR-4). Расход УВОДИТ количество из системы (`to=null`).

**5. Деньги — живой пересчёт, ничего не хранится.** Принятый вес одноступенчато (`computeAcceptedKg`).
Два независимых гейта (BR-10): `is_accepted` → `accepted_weight`; `contract_line_id != null` → оплата/выполнение.

**6. Колонки-снимки `accepted_weight_kg`/`brak_weight_kg` — DEPRECATED**, считаются на лету. ⚠️ Физически
есть в схеме, но НЕ пишутся (см. ACTIVE RISKS).

**7. Единый складской движок `StockMovement` (kind packaging|ingredient).** Баланс = Σ движений по
(локация × предмет × состояние). Лом — состояние, не локация (только у тары; ингредиент всегда good).

**8. Доставка ингредиента — ТОТ ЖЕ `MaterialShipment`, параметризованный по `item_kind` (вариант A, E1).**
Один движок строит плечи построчно по `kind`+FK; **смешанный груз (тара + ингредиент в одном рейсе) —
штатный кейс**. Ингредиент использует только транзит `-2` (на завод `-1` не ходит). Дашборд ингредиентов —
ЗЕРКАЛО дашборда тары (`getIngredientBalances`), а не параметризация (отличия структурные: нет scrap,
один транзит, единицы кг/л).
- Не менять без обсуждения: единый движок (не плодить ingredient-копии плеч); транзит ингредиента только `-2`.

**9. Прибытие доставки — по-позиционное; статус «частично» производный.** Источник истины —
MaterialShipmentItem.arrived_at; enum ShipmentStatus (общий с продукцией) НЕ расширяется. Дубли позиций
`(farmer, kind, fk)` в рейсе запрещены (zod) вместо item-link в леджер (YAGNI: дубль = одна строка с суммой).
Не менять: отправка whole-trip; «частично» не хранить; гард отката при прибывших.

# INVARIANTS
- Всегда вычислять балансы/стоимость/принятый вес (НЕ хранить).
- Никогда не писать `accepted_weight_kg` / `brak_weight_kg` (deprecated).
- Баланс склада = Σ `StockMovement` по (локация × предмет × состояние).
- Локации: `0`=завод · `Farmer.id` · `-1`=в пути на завод · `-2`=в пути с завода · `null`=вне системы.
- Движение тары/доставки двухфазное: дебет при отправке (→транзит), кредит при прибытии (транзит→получатель).
- **Ингредиент: транзит только `-2`; состояние всегда good (нет лома); расход (`consumption`) уводит из
  системы (`to=null`); единицы кг/л не суммируются между собой.**
- Принятый вес — одноступенчато через `computeAcceptedKg`; никакого «годного» = факт×(1−брак).
- Два гейта: `is_accepted` → accepted_weight; `contract_line_id` → оплата/выполнение.
- Ингредиент: расход по ФАКТИЧЕСКОМУ весу.
- Source движений не путать: `shipment` (вход) vs `material_shipment` (доставка) vs `acceptance_act` (расход).
- Единый движок: НЕ плодить ingredient-копии — параметризовать по `kind` (где отличие — дискриминатор).
- **Лом — СОСТОЯНИЕ, не локация:** scrap = loc/good→loc/scrap (та же локация); disposal = loc/scrap→null.
- **adjustment-инвентаризация:** движение на разницу (факт − текущий Σ); причина операций тары — в ChangeLog.
- **Прибытие доставки — per-item** (`arrived_at`); «частично» производное (не хранить); дубли позиций
  `(farmer, kind, fk)` в рейсе запрещены; отправка whole-trip; откат sent→planned только при 0 прибывших.
- Эталоны: CRUD — Farmer; матрица — NormsMatrix/OpeningStock; нетто-сторно/идемпотентность — materials/movements.ts.
- Логика — в `server/`; компоненты только отображают и зовут server-actions.
- Схему и доки менять только через обсуждение; тонкие вертикальные срезы; context7 для API библиотек.
- **Dev/staging БД — тестовая, данные одноразовые** (см. CLAUDE.md): миграции/рефакторинги могут свободно
  пересоздавать данные; перед продом БД стирается.
